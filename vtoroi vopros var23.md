Что такое объектно-реляционное отображение (ORM)? Чем ваш самописный уровень доступа отличается от готовых ORM вроде Eloquent или Doctrine?
Определение ORM

ORM (Object-Relational Mapping) — это технология программирования, позволяющая связывать реляционные базы данных с объектно-ориентированными языками, такими как PHP. Вместо написания SQL-запросов разработчик работает с объектами, а ORM автоматически преобразует операции над объектами в SQL-запросы и обратно.

Простыми словами: строка из таблицы Client превращается в объект $client, у которого есть свойства ($client->full_name) и методы ($client->save(), $client->delete()).
Пример работы с ORM (на примере Eloquent из Laravel)
php

// Поиск клиента
$client = Client::find(1);

// Изменение данных
$client->full_name = 'Иванов Иван Иванович';
$client->phone = '+79991234567';

// Автоматическая генерация UPDATE
$client->save();

// Получение связанных записей (Lazy loading)
$appointments = $client->appointments;

Чем мой самописный DAL отличается от ORM

В ходе выполнения работы был разработан собственный уровень доступа к данным (DAL) на основе паттерна Repository, а не готовая ORM-библиотека. Основные отличия представлены в таблице.
Характеристика	Мой DAL (Repository)	ORM (Eloquent / Doctrine)
Формат возвращаемых данных	Ассоциативные массивы	Объекты с методами
Необходимость написания SQL	Да, внутри методов репозиториев	Нет, SQL генерируется автоматически
Работа со связями таблиц	Ручное написание JOIN	Автоматическая (lazy / eager loading)
Объём кода	Больше (каждый метод пишется вручную)	Меньше (достаточно описать свойства)
Гибкость и контроль	Полный контроль над каждым запросом	Ограничен возможностями ORM
Производительность	Выше (выполняется только явно написанный код)	Ниже (возможны лишние или неоптимальные запросы)
Порог вхождения	Требуется знание SQL	Ниже, так как SQL не нужен
Прозрачность	Полная (SQL всегда виден разработчику)	Частичная (ORM может генерировать неожиданные запросы)
Сравнение на конкретном примере

Мой DAL (получение клиента с его записями):
php

// Явное написание JOIN
$sql = "SELECT c.*, a.app_datetime, a.status 
        FROM Client c
        LEFT JOIN Appointment a ON c.client_id = a.client_id
        WHERE c.client_id = :id";
$stmt = $db->prepare($sql);
$stmt->execute([':id' => 1]);
$result = $stmt->fetchAll(); // возвращает массив

ORM (Eloquent) при аналогичной задаче:
php

// Описание связи в модели
class Client extends Model {
    public function appointments() {
        return $this->hasMany(Appointment::class);
    }
}

// Использование
$client = Client::find(1);
$appointments = $client->appointments; // SQL генерируется автоматически

Почему выбран самописный DAL, а не готовая ORM

Для данной учебной работы использование самописного уровня доступа к данным было более целесообразным по следующим причинам.

    Образовательная цель: требовалось продемонстрировать понимание работы с базами данных на низком уровне, включая написание SQL-запросов, работу с PDO и подготовленными выражениями.

    Полный контроль: каждый SQL-запрос явно написан и может быть оптимизирован под конкретную задачу.

    Отсутствие "магии": код предсказуем, легко отлаживать и анализировать.

    Лёгкость: не требуется подключать и настраивать тяжёлые ORM-библиотеки (Doctrine весит десятки мегабайт).

    Соответствие заданию: требовалось именно «разработать уровень доступа к данным (DAL)», а не «использовать готовый ORM».

Недостатки самописного DAL по сравнению с ORM

В реальной коммерческой разработке самописный DAL имеет ряд ограничений.

    Большой объём шаблонного кода для каждой новой таблицы.

    Необходимость вручную писать SQL для операций со связями.

    Отсутствие автоматического преобразования строк в объекты.

    Нет встроенных механизмов кеширования и событий (события до/после сохранения).
